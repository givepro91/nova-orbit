# 작업 요약 투명성 — Before/After 서사 + 스크린샷 (있으면)

> 상태: 설계 승인됨 (2026-07-09), **구현 완료**.
> ⚠ **2026-07-20 개정됨 — 아래 [§개정 2](#개정-2-2026-07-20--요약이-diff를-읽는다) 를 먼저 읽을 것.** 본문 §1의 요약 입력 정의와 §10-17의 "코드 diff 뷰어 기각"이 뒤집혔다. 나머지 기각 항목(headless 렌더 · 스크린샷 의무화 · before/after 페어링)은 유효하다.
> 문제 제기: "에이전트에게 다 맡기려니 신뢰가 안 간다 — 특히 비주얼 작업을 어떻게 했고, 정확히 뭘 바꿨고, 요약이 뭔지 디테일이 부족하다."

## 목표 (한 줄)

goal이 끝났을 때, 사람이 **5초에** "원래 이랬는데 → 이 작업으로 → 이렇게 됐다"를 파악하는 **before/after 서사 요약**을 보여준다. 작업 중 자연히 생긴 **스크린샷이 있으면** 그 옆에 곁들인다. 스크린샷은 어디까지나 옵션이다.

## 왜 이 형태인가 (기각한 대안)

신뢰의 순간은 goal squash 승인 게이트("이걸 main에 맡길까?")다. 그 화면이 지금은 **파일명 목록 + 커밋 메시지**에서 멈춘다 (`GoalSquashApprovalDialog.tsx`). 사람이 읽고 판단할 서사가 없다.

검토 끝에 아래를 **명시적으로 범위에서 뺀다** — 취약하거나 토큰/시간 낭비 대비 신뢰 이득이 낮다:

- ❌ **Crewdeck이 headless로 before/after를 직접 렌더** — 대부분 실앱은 인증·백엔드·시드데이터 없이는 바 루트에서 의미 있는 화면이 안 나온다("로그인 vs 로그인" = 쓸모없는데 안심시키는 최악 신호). dev 서버를 워크트리 2개에서 기동·포트·준비감지·teardown 하는 취약성 + 스크린샷 노이즈까지. 신뢰 이득 대비 비용 과다.
- ❌ **코드 diff(hunk) 뷰어** — 사용자 판단: 라인 단위 수정보다 "무엇이 어떻게 됐나" 서사가 우선.
- ❌ **에이전트에 스크린샷 의무화** — 강제하는 순간 작업 시간·토큰만 늘고, 안 찍어도 되는 작업까지 찍게 된다.

## 구성요소

### 1. Before/After 작업 요약 (핵심)

**goal 완료 시점(squash-ready)에 값싼 모델 1콜**로 goal 누적 맥락을 받아 사람용 서사를 합성한다.

- **입력**: 완료 태스크들의 `result_summary` + 변경 파일 목록(`git diff --name-only <base>...HEAD`, 이미 `engine.ts` squash 경로에서 수집) + goal 제목/설명 + 커밋 메시지. 입력 크기 상한을 둔다(토큰 방어).
- **출력(구조화 JSON)**:
  - `before` — 작업 전 상태/문제 (이 goal이 풀려던 것)
  - `changed` — 무엇을 했나 (기능·화면·동작 단위, **파일 라인 아님**)
  - `after` — 지금 어떻게 달라졌나 / 사용자가 보게 될 차이
  - `notes` — (선택) 주의점·미해결
- **왜 goal 레벨 합성 콜인가**: goal은 여러 태스크·에이전트에 걸쳐 있어 어느 한 에이전트도 전체 그림을 모른다. goal당 **딱 1콜**이라 비용은 무시 가능(신뢰 판단 1건에 값싼 모델 1콜).
- **생성 실패는 게이트를 막지 않는다**: 요약 콜이 실패하면 현재처럼 커밋메시지+파일목록으로 폴백(요약 섹션만 비고 승인은 정상 진행). 요약은 부가정보이지 blocker가 아니다.

### 2. 태스크 레벨 요약 (추가 콜 0)

지금 `engine.ts:854`가 에이전트 최종 텍스트를 **뒤 500자로 자른다**(`MAX_SUMMARY_LEN`) — 문장 중간에서 잘려 읽기 나쁘고 UI 노출도 거의 없다. 이를 **에이전트가 마지막에 자연히 쓰는 마무리 텍스트 블록**(stream-parser의 누적 text에서 마지막 assistant 메시지)을 온전히 담도록 바꾼다. **추가 LLM 콜 없음** — 이미 나온 출력을 제대로 담는 것뿐. 태스크 상세에 노출.

> ⚠ `result_summary`는 컨텍스트 체인 주입에도 쓰인다(`engine.ts:717-726`, 앞 200자). 담는 내용을 바꾸되 이 소비처가 깨지지 않게 유지(길이 상한 별도 관리).

### 3. 스크린샷 — 있으면 수집·표시 (제로 강제)

- goal 완료 시 워크트리를 **읽기 전용 스캔**: 알려진 캡쳐 산출 경로(`.playwright-mcp/`, `.cc-shots/`)와 에이전트가 저장한 이미지(`*.png`/`*.jpg`)를 찾는다.
- 있으면 `~/.crewdeck/artifacts/goals/<goalId>/`로 복사해 artifact로 등록, 요약 옆 썸네일로 노출. 없으면 섹션 자체가 안 뜬다.
- evaluator가 `.playwright-mcp`/`.cc-shots`를 평가 diff에서 "노이즈"로 제외하는 로직(`evaluator.ts` `TOOL_STATE_PATHS`)은 **그대로 둔다**(평가 방해 방지 목적). 수집은 그와 무관한 별도 read-only 경로.

# 개정 2 (2026-07-20) — 요약이 diff를 읽는다

> 상태: **구현 완료 (2026-07-20)**. 위 원설계(2026-07-09)를 폐기하지 않고 **입력과 근거 접근만** 개정한다.
> 미검증으로 남은 것: 실제 goal을 squash까지 돌려 다이얼로그를 눈으로 확인하는 절차(아래 §테스트 마지막 항목)와,
> LLM이 실제로 diff를 읽고 `outOfScope`를 뽑아내는지. 프롬프트가 전달되는 지점까지만 검증됐다.

## 계기 — 서사만으로는 못 잡은 사례

`swk-infra-console`의 한 goal이 완료됐다. 요청은 "목록↔상세 스펙 모순 2줄 수정"이었고 실제로 그 2줄은 정확히 고쳐졌다. 그런데 같은 diff에 **요청하지 않은 머신 6대 alias 재벤더링(+38/-9줄)** 이 codegen 재실행으로 함께 딸려 들어왔다. 데이터는 정확했지만 커밋 성격이 달랐다 — 승인자가 "따로 커밋할까"를 판단해야 하는 정보였다.

사용자가 이 사실을 안 것은 **터미널에서 에이전트가 자발적으로 말해줬기 때문**이고, 승인 다이얼로그만 봤다면 보지 못했다. 구조적으로 나올 수 없는 정보였기 때문이다 — 원설계 §1의 요약 입력이 `result_summary`(에이전트 자기 보고) + **파일 이름 목록**뿐이라, "그 파일 안에서 무엇이 얼마나 바뀌었나"는 애초에 입력에 없었다.

이것은 구현 버그가 아니라 **설계 가정의 한계**다. 그리고 Generator-Evaluator 분리 원칙(구현자와 검증자는 항상 다른 세션)이 요약 경로에서만 지켜지지 않고 있었다는 뜻이기도 하다 — 요약은 산출물이 아니라 *구현자의 주장*을 재요약하고 있었다.

## 무엇을 뒤집고, 무엇을 유지하나

| 원설계 | 개정 | 근거 |
|---|---|---|
| §1 입력 = task 요약 + **파일 목록** | 입력에 **diff 실체**(`--stat` + 예산 내 hunk) 추가 | 위 사례. 자기 보고의 재요약으로는 범위 이탈을 못 잡는다 |
| ❌ 코드 diff 뷰어 (§17) | **2차 근거로 부활 (기본 접힘)** | 원 기각 논거는 "라인 수정보다 서사가 **우선**"이라는 *우선순위* 판단이지 배제가 아니었다. 서사가 1차 화면이고 diff가 접힌 근거면 그 판단과 양립한다. 게다가 기각 당시엔 없던 `GET /goals/:goalId/diff` + `DiffPane.tsx`가 그 사이 세션 워크스페이스용으로 생겨 배선 비용이 거의 0이다 |
| ❌ headless before/after 렌더 | **기각 유지** | 인증·시드데이터 문제 그대로. 크루덱에 playwright/puppeteer 의존성이 0인 것도 그대로 |
| ❌ 스크린샷 의무화 | **기각 유지** | "제로 강제" 원칙 유지 |
| ❌ before/after 스크린샷 페어링 | **기각 유지** | baseline 캡처는 크루덱이 브라우저를 제어해야 가능 — 곧 위 기각 항목과 같은 것이다 |

"사용자가 보게 될 차이"를 강화하려던 요구는 **스크린샷 짝이 아니라 요약 스키마의 구조화**로 푼다(아래 §요약 스키마). 새 인프라 0.

## 요약 스키마 (§1 출력 개정)

```json
{
  "commitType": "feat|fix|update|docs|refactor|chore|test",
  "before": "작업 전 상황/문제 (1-2문장)",
  "changed": "무엇을 했는지 (2-4문장, 기능·화면·동작 단위)",
  "after": "지금 어떻게 달라졌는지 (한 문장)",
  "userImpact": {
    "visible": true,
    "surfaces": [{ "name": "화면·엔드포인트·명령어 이름", "change": "무엇이 달라지나" }]
  },
  "outOfScope": "goal이 요청하지 않았는데 diff에 포함된 변경 (없으면 빈 문자열)",
  "notes": "주의점·미해결 (없으면 빈 문자열)"
}
```

두 필드가 새로 생긴다.

- **`userImpact`** — 원설계의 `after` 한 문장을 구조화한다. 변경 유형에 적응한다: UI면 화면 단위, API·CLI면 인터페이스 단위. 내부 리팩터라 사용자 체감이 없으면 `visible: false` + 빈 `surfaces`로 **"없음"을 명시적으로 선언**한다(침묵과 구별되어야 한다). `after`는 유지하되 한 문장 헤드라인 역할로 좁힌다.
- **`outOfScope`** — 이번 개정의 핵심. goal 범위 밖 변경이 섞였는지를 요약이 **먼저 말하게** 한다. 승인자가 diff를 정독해야만 발견할 수 있던 것을 1차 화면으로 끌어올린다.

## diff 주입 방식

- **세션은 계속 격리 temp dir에서 스폰한다.** 원설계가 goal worktree를 피한 이유(승인 시 worktree `--force` 제거와 도구 사용 가능한 서브프로세스가 겹치는 창)는 그대로 유효하다. 대신 **서버가 diff 텍스트를 미리 읽어 프롬프트에 주입**한다 — 세션은 여전히 파일시스템에 접근하지 않는다.
- **예산**: `git diff --stat <base>...HEAD` 전문은 항상 넣고(전체 형상 파악용), hunk 본문은 상한까지만 넣는다. 기존 diff 엔드포인트의 500KB 캡은 사람이 보는 뷰어용이라 프롬프트엔 과하다 — 별도의 더 작은 상한을 둔다.
- **노이즈 제외**: lock 파일, `.playwright-mcp/`, `.cc-shots/` 등. `evaluator.ts`의 `TOOL_STATE_PATHS`가 이미 같은 목적의 목록을 갖고 있으므로 재사용을 검토한다(단, evaluator의 것은 평가 방해 방지용이라 목적이 달라 그대로 공유할지는 구현 시 판단).
- **⚠ truncate 정직성**: hunk가 예산에 잘렸으면 그 사실을 프롬프트에 명시하고, 요약이 `outOfScope`를 **단정하지 않게** 한다. "전부 봤다"고 착각한 요약이 "범위 밖 변경 없음"이라고 단언하는 것이 이 개정에서 가장 위험한 실패 모드다 — 없다고 말해서 안심시키느니 불확실을 드러내야 한다.

## UI (§UI 개정)

`GoalSquashApprovalDialog.tsx` — 1차 화면은 여전히 서사다. 목업으로 확인한 확정 배치는 아래와 같다.

**섹션 순서** — 서사가 맥락을 주고 → 구체화하고 → 경고가 뜬 뒤, 근거가 따라온다:
목표 · 반영 브랜치 → 작업 요약(서사) → **사용자가 보게 될 차이**(+스크린샷) → **요청하지 않은 변경** → 커밋 본문 → 변경 파일 → 검증 결과 → 품질 게이트 → 코드 변경.
경고를 승인 버튼 근처가 아니라 상단부에 두는 것은 의도적이다 — 스크롤로 놓치지 않게 한다.

- **`userImpact` 블록** — 서사에 이어 "사용자가 보게 될 차이"를 surface 단위 행으로. `visible: false`면 "사용자 체감 변화 없음"을 같은 자리에 **명시**한다(침묵과 구별).
- **스크린샷은 이 블록 바로 아래로 이동**한다(원설계에선 서사 하단). 스크린샷은 본질적으로 `userImpact`의 증거이고, 서사 하단에 두면 텍스트 증거와 시각 증거가 갈라진다. 단 `collectScreenshots`가 `taskId`를 항상 `null`로 채우므로(`work-report.ts:139`) surface와 1:1로 묶지는 못한다 — 섹션 이동까지만.
- **`outOfScope` 경고 블록** — 비어 있을 때만 숨긴다. 톤은 **warning(주황) 고정, danger 아님**. codegen 재실행·포매터·lock 파일 재생성처럼 정상적으로 딸려오는 경우가 흔해 매번 빨강이면 늑대소년이 된다. danger는 이미 `blocked`(게이트가 막힘)가 점유하고 있고, 이 신호의 의미는 "멈춰라"가 아니라 **"판단해라"** 이므로 warning이 의미론적으로 맞다. 기존 `skippedTasks`와 같은 위계.
- **범위 밖 파일을 목록에서 강조하지 않는다.** 변경 파일 목록에 색점을 찍는 안을 검토했다가 기각했다 — `outOfScope`는 서술 문자열이라 파일을 지목할 구조가 없고, 지목하게 만들면 **엉뚱한 파일이 범인이 되어 근거 없는 의심**을 만든다. 대신 프롬프트에서 관련 경로를 서술 안에 자연스럽게 쓰게 한다(필드를 늘리지 않고 같은 정보를 얻는다).
- **접이식 "코드 변경 보기"** — **항상 접힘**. 이것만이 진짜 2차 근거다. 펼치면 기존 `GET /goals/:goalId/diff` + `DiffPane.tsx` 재사용 — 새 git 호출도 새 렌더러도 만들지 않는다.
- **검증 결과 칸 실동작화** — 지금 이 섹션은 구조적으로 절대 렌더되지 않는다: 엔진이 `acceptanceOutput: ""`를 하드코딩하고(`engine.ts:3824`), `squash-preview` 응답엔 그 필드 자체가 없다(`goals.ts:1338-1346`). 실제 acceptance 실행 출력을 채우고 preview 응답에도 싣는다. 원설계 §55가 "인접 개선 — 선택"으로 남겼던 항목의 회수다.
- **quality-gate 이슈** — 지금은 커밋 메시지의 `통과 N · 조건부 M · 실패 K` 한 줄로만 축약된다. 제목 + evidence를 노출하되, **조건부·실패가 하나라도 있으면 펼친 상태, 전부 통과면 접힘**으로 연다. 조건부·실패는 "의심이 갈 때 내려가는 근거"가 아니라 승인자가 이미 알아야 할 판정이기 때문이다. 데이터는 `verification-timeline` API에 이미 있다.

> 목업(2026-07-20): 실제 사례 데이터로 위 배치를 재현해 확인했다. 라이트·다크 양쪽 토큰은 `index.css`를 그대로 따른다.

## 후속 — goal 상세의 "자세히 이해하기" (이번 범위 밖)

승인 게이트는 "**main에 넣어도 되나**"를 30초~2분에 판단하는 자리다. "이 코드를 이해했나"는 다른 질문이고, 다른 화면이 필요하다. `/nova:explain-diff`류의 해설(배경 → 직관 → 코드 워크스루 → 이해도 퀴즈)은 **승인 다이얼로그에 통합하지 않고**, `GoalDetail.tsx` 헤더의 액션으로 opt-in 배치한다(현재 이 파일엔 뮤테이션 액션 버튼이 하나도 없어 우측 슬롯이 비어 있다).

외부 스킬에 의존하지 않는다 — 사용자마다 설치 여부가 다르므로 크루덱이 자체 프롬프트로 생성한다. 산출물 형태(대시보드 내 패널 vs 포터블 HTML)와 비용 모델은 별도 설계로 다룬다.

## 오류/degrade (원설계 원칙 유지)

- diff 읽기 실패 → 파일 목록만으로 기존 경로 폴백. **게이트 블로킹 금지.**
- 요약 콜 실패 → 원설계대로 섹션 생략, 승인 정상 진행.
- `userImpact`/`outOfScope` 파싱 실패 → 해당 블록만 생략하고 나머지 서사는 표시(스키마 확장이 기존 요약을 깨뜨리지 않게 파서는 관대하게).
- diff 엔드포인트 실패 → 접이식 섹션에 오류 표시. 승인 자체는 가능.

## 테스트

- diff 주입 빌더: 예산 내/초과, 노이즈 제외, truncate 플래그가 프롬프트에 실리는지 유닛.
- 확장 스키마 파서: 신규 필드 정상/누락/타입 불일치 → 관대한 폴백 유닛.
- `acceptanceOutput`이 broadcast와 preview 응답 양쪽에 실리는지 (지금 항상 빈 문자열인 회귀의 재발 방지).
- 다이얼로그: `outOfScope` 비었을 때 블록 미표시, `visible:false`일 때 "체감 변화 없음" 표시 (dashboard vitest).
- ⚠ 실제 진입점 검증: 유닛 GREEN이 아니라 **goal 하나를 끝까지 돌려 승인 다이얼로그를 눈으로 확인**한다.
