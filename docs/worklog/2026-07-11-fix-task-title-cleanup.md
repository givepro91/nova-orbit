# Worklog — 자동 수정 작업의 제목 정리 · 2026-07-11

## 요약 (TL;DR)
- **문제**: 자동으로 만들어지는 "수정 작업"들의 제목이 목록에서 무엇을 고치는 작업인지 알아볼 수 없을 만큼 이상하게 나왔다.
- **한 일**: 수정 작업 제목을 원래 작업 이름 기반으로 정리하고, 한 번의 검사에서 수정 작업이 무제한으로 쏟아지지 않게 개수 상한을 넣었다.
- **지금**: 새로 만들어지는 제목이 깔끔해졌고 이미 쌓여 있던 제목도 일괄 정리했다. 자동 테스트 521개가 모두 통과했고 라이브 서비스에 반영됐다.
- **남은 결정**: 사용자가 본 "문법 오류"의 정확한 위치를 알려주면 이어서 확인한다.

## 한 일 (What we did)
품질 검증(Quality Gate, 구현물을 다른 세션이 채점하는 단계)이 실패하면 문제마다 "수정 작업"이 자동 생성된다. 그 제목이 `[Fix][dataFlow] <장황한 진단 문장…>` 형태로, 실패한 검증 항목의 영문 이름과 진단 텍스트를 문장 중간에서 잘라 붙인 것이었다. 대시보드 목록에서는 이게 무슨 작업인지 읽어낼 수 없었다.

제목을 **분할(decompose)로 만든 원래 작업의 이름**을 재사용하도록 바꿨다. 실패한 검증 항목만 한국어 라벨로 덧붙여 `[수정] Goal Spec 생성·편집·승인 API · 데이터 흐름`처럼 읽히게 했다. 사용자가 "무한 루프에 빠지는 것 아니냐"고 물어, 수정 루프에 걸린 안전장치를 코드로 확인했다. 라운드 상한과 진전 없음 감지, 재시도 소진 후 영구 정지까지 3중으로 막혀 있어 무한은 아니었다. 다만 검증 한 번이 뱉는 문제 개수만큼 수정 작업이 제한 없이 생성되던 빈틈이 있어, 검증당 상한(`MAX_FIX_TASKS_PER_VERIFICATION`, 기본 5)을 심각도 높은 순으로 추가했다.

배포 후 사용자가 두 번째 스크린샷으로, 일부 제목이 여전히 `[수정] [Fix][designAlignment] …`처럼 서술형으로 겹쳐 나오는 걸 지적했다. 원인은 수정 작업이 또 실패해 만들어진 "수정의 수정"(중첩)이었다. 이때 바로 위 원본이 다시 수정 작업이라 그 서술형 제목이 그대로 base로 쌓였다. `resolveRootTaskTitle`을 추가해 수정 작업의 원인 사슬을 거슬러 올라가 **맨 처음 분할 작업**의 이름까지 도달하게 했다. 이제 중첩이 아무리 깊어도 제목은 항상 `[수정] <원래 작업 이름> · <검증 항목>`이 된다.

이미 데이터베이스에 쌓여 있던 옛 제목은 코드만 고쳐선 안 바뀌므로, 저장된 작업 36개를 API로 일괄 재명명했다. 그중 23개가 중첩이라 사슬을 거슬러 올라가 원래 이름으로 계산해 넣었다.

## 결정과 왜 (Decisions & why)
| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 제목을 진단 텍스트 대신 원래 작업 이름 기반으로 생성 | 실패 진단문(`issue.evidence`)을 잘라 쓰던 기존 방식 | 진단문은 길고 문장 중간이 잘려 목록에서 무슨 작업인지 식별이 불가능했다. 원래 작업 이름은 분할 단계에서 사람이 읽기 좋게 지어진 이름이라 그대로가 낫다. |
| 중첩 수정은 원인 사슬을 거슬러 올라가 근본 작업 이름을 찾음 | 제목 앞의 `[수정]` 접두사만 문자열로 제거 | 접두사만 벗기면 중첩된 옛 제목의 서술형 진단 텍스트가 그대로 남는다. 데이터베이스의 관계를 따라 근본 작업까지 올라가야 서술형 누적이 원천 차단된다. |
| 검증당 수정 작업 개수를 심각도 상위 N개로 제한 | 개수 무제한 / 목표(goal)당 총량 제한 | 한 검증이 문제를 10여 개 뱉으면 목록이 그만큼 불어났다. 심각도 순 상한이 가장 단순하면서 중요한 문제를 우선 보존한다. |
| 내 작업과 무관한 기존 실패 테스트 2건도 고치고 배포 | 무관하다는 이유로 그대로 두고 배포 | 원래 브랜치에서도 동일하게 실패하던 오래된 단언 불일치였다. 사용자가 "테스트 통과 시 배포"를 기준으로 삼았으므로 스위트 전체를 통과 상태로 만드는 편이 기준에 맞았다. |
| 재배포 시 실행 중인 에이전트 세션을 죽이지 않고 자연 종료를 기다림 | 즉시 재시작 | 실행 중 세션을 강제 종료하면 작업이 유실된다(과거 실측). 큐를 멈춰 새 작업 유입만 막고 진행 중 2개가 끝날 때까지 9분을 기다려 유실 없이 배포했다. |

## 배운 것 (Learned)
- 수정 작업의 제목은 바로 위 원본이 아니라 **근본 원본까지 거슬러 올라가** 지어야 중첩에서 무너지지 않는다. 파생 데이터의 이름은 항상 가장 가까운 부모가 아니라 뿌리를 참조해야 한다는 일반 원칙이다.
- 이 프로젝트의 빌드는 서버 코드를 `dist/server/index.js`에 번들한다. 실행 진입점인 `dist/bin/crewdeck.js`가 그걸 불러 쓰는 구조라, 배포 검증 때 진입점 파일만 grep 하면 변경이 없는 것처럼 보인다.
- 큐를 멈춘 뒤에도 겉으로 멈춰 보이지 않는 세션은, 외부 도구의 재접속 재시도 한도가 소진되면 결국 스스로 끝난다. 무작정 강제 종료하기 전에 한 번 더 기다려 볼 값어치가 있다.

## 세션 흐름 (How it moved)
1. 첫 스크린샷 제보로 제목이 `[Fix][dim] 진단문` 형태임을 확인하고 원인을 `engine.ts`의 제목 생성 한 줄로 좁혔다.
2. 제목 빌더를 원래 작업 이름 기반으로 바꾸고, 검증당 수정 작업 개수 상한을 추가했다.
3. "무한 루프" 우려에 답하려 수정 루프의 안전장치 3종을 코드로 확인했다.
4. 지침대로 작업을 별도 워크트리로 격리하도록 바로잡았다(초기에 main에서 직접 편집한 것을 되돌림).
5. 무관한 기존 실패 테스트 2건을 확인·수정해 전체 521개를 통과시켰다.
6. 1차 배포: main 병합 후 자연 drain, 빌드, 재시작.
7. 저장돼 있던 옛 제목 33개를 API로 재명명했다.
8. 사용자 요청으로 crewdeck 프로젝트를 반자동 모드로 켰다.
9. 두 번째 스크린샷으로 중첩 제목 문제를 제보받아 근본 작업 추적 로직을 추가했다.
10. 중첩 제목 23개를 재명명하고 코드 수정을 2차 배포했다.

## 남긴 것 (Left open)
- 사용자가 "문법 오류가 발생한다"고 알렸는데, 서버 로그와 재빌드한 화면 번들에서는 현재 재현되는 문법 오류가 없다. 로그에 남은 것은 2~3일 전 에이전트가 반환한 JSON 파싱 실패뿐이고, 오늘 나는 것은 외부 도구(Codex)의 접속 시간 초과다. 사용자가 본 정확한 화면이나 메시지를 알려주면 이어서 확인해야 한다.
- 수정 작업 개수 상한을 검증당이 아니라 목표(goal)당 총량으로도 걸지는 보류했다. 지금은 필요성이 확인되지 않았다.

## 결과·검증 (Outcome)
서버 타입 검사가 통과하고, 자동 테스트 521개가 모두 통과했다. 라이브 서비스는 재빌드 후 재시작해 정상 응답하며(health ok), 실행 중인 번들에 새 로직이 포함된 것을 확인했다. 저장된 제목 데이터는 서술형 잔재 없이 정리됐고, 다시 만들어진 화면 번들도 문법 검사를 통과했다. 이번 세션은 `/gate`(정직성 검증 단계)를 돌리지 않았으므로, 위 결과는 각 단계에서 직접 실행해 관찰한 값이다.
