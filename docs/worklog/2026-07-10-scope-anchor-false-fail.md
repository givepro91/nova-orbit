# Worklog — 검토 단계 억울한 실패의 진원지를 찾아 없앤 날 · 2026-07-10

## 요약 (TL;DR)
- **문제**: 앱이 스스로 만든 결과물을 다시 검사하는 "검토" 단계에서 실패가 너무 자주 떴다. 사용자가 "왜 이렇게 자꾸 문제가 생기도록 개발하느냐"고 물었다.
- **한 일**: 실패의 진짜 원인을 실제 데이터로 갈라, 그중 "잘 만들었는데도 떨어지는" 억울한 실패를 만들던 규칙 하나를 고쳤다.
- **지금**: 검사 기준이 더는 파일 위치를 강제하지 않는다. 올바른 구현이 위치 차이만으로 떨어지지 않게 됐고, 상시 서비스에 반영·재가동까지 마쳤다.
- **남은 결정**: 없음. 진짜 결함을 잡아내는 검사의 엄격함은 그대로 두었다.

## 한 일 (What we did)
검토에서 실패가 쏟아지는 것이 "구현이 나빠서"인지 "검사가 너무 빡세서"인지부터 갈랐다. 상시 서비스의 실제 검증 기록을 열어 세 건의 실패 전문을 읽었다. 대부분은 검사기가 실제로 코드를 실행해 잡아낸 **진짜 결함**이었다. 정상 실행마다 로그가 새는 문제, 계약을 지킨다고 해놓고 실제로는 강제하지 않는 코드, 서로 다른 작업이 살아있는 세션을 죽이는 경합이 그것이다. 검사기는 고장 난 게 아니라 오히려 유능하게 일하고 있었다.

문제는 다른 한 축에 있었다. 작업을 잘게 쪼갤 때 계획 단계가 "이 작업은 이 파일들을 건드릴 것"이라고 파일 목록을 **추측**해서 박는다. 그런데 이 추측이 구현 단계에서는 "오직 이 파일만 고쳐라"로 강제되고, 검사 단계에서는 "예상 파일이 안 보이면 실패"로 채점된다. 추측이 실제 코드 구조와 어긋나면, 올바른 위치를 고친 구현조차 "범위 이탈"로 떨어진다.

그래서 이 파일 목록을 **계약이 아니라 안내**로 격하했다. 세 지점(계획을 세우는 곳, 구현을 지시하는 곳, 검사하는 곳)을 일관되게 손봤다. 같은 기능을 다른 경로에 구현한 것은 정답으로 인정하고, 전혀 다른 기능이나 엉뚱한 기술로 벗어난 진짜 이탈만 실패로 남겼다.

수정을 상시 서비스에 반영하는 과정에서 곁가지 문제도 둘 정리했다. 테스트 하나가 실행 환경 설정에 오염돼 실패하던 것을 격리했고, 이번 판단의 전말을 로드맵과 에이전트 규칙 파일에 기록했다.

## 결정과 왜 (Decisions & why)
| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 파일 목록을 계약에서 안내로 격하 | 검사 기준 전체를 느슨하게 완화 | 억울한 실패는 오직 "위치 강제" 한 곳에서 나왔다. 검사 전체를 풀면 진짜 결함까지 놓친다. 문제의 원인만 정확히 도려냈다. |
| 진짜 결함을 잡는 검사 엄격함은 그대로 유지 | 실패율 자체를 낮추도록 검사를 무르게 | 실패가 많은 것 자체는 나쁜 신호가 아니다. 검사기가 테스트로 못 잡는 결함을 실제 실행으로 잡아내는 것은 이 도구의 핵심 가치다. |
| 정석 절차 대신 서비스를 곧바로 재시작 | 실행 중 작업이 자연히 끝날 때까지 대기 | 큐를 멈춰도 진행 중이던 작업이 새 작업을 계속 이어 물어, 자연 종료가 오지 않았다. 사용자가 "지금 중단 수용"을 이미 택했기에 무한 대기보다 즉시 반영이 옳았다. |
| 에이전트 규칙에 이 레포 실제 관행을 커스텀으로 기록 | 도구가 주는 일반 템플릿을 그대로 삽입 | 일반 템플릿은 "main에 직접 커밋 금지"를 강제하는데, 이 레포는 main에 직접 커밋하는 개인 도구다. 그대로 넣으면 앞으로 에이전트를 오히려 오도한다. |

## 배운 것 (Learned)
- 검사기가 파일을 읽는 데 그치지 않고 **실제로 실행**하면, 테스트는 초록불인데 실제 동작은 깨지는 갭이 드러난다. 실패가 늘어 보여도 이건 강점이다.
- 언어 모델의 추측을 계약으로 승격시키면 안 된다. 추측은 안내로 두고, 실제 코드가 다르면 코드를 신뢰하게 해야 한다.
- 테스트가 개발자 각자의 전역 설정(예: 전역 `gitignore`)에 오염될 수 있다. 임시 저장소를 쓰는 테스트는 환경을 명시적으로 차단해 독립성을 지켜야 한다.

## 세션 흐름 (How it moved)
1. 실패가 "나쁜 구현" 탓인지 "빡센 검사" 탓인지 실제 검증 기록으로 갈랐다.
2. 대부분은 진짜 결함, 한 축만 억울한 실패임을 확인했다.
3. 억울한 실패의 진원지가 "추측한 파일 목록의 강제"임을 코드로 특정했다.
4. 계획, 구현, 검사 세 지점을 계약에서 안내로 일관되게 고쳤다.
5. 큐를 멈추고 서비스를 다시 빌드해 재시작하는 방식으로 상시 서비스에 반영했다.
6. 곁가지로 테스트 환경 오염을 격리하고, 로드맵과 에이전트 규칙에 기록했다.

## 남긴 것 (Left open)
- 실제 재검증에서 억울한 실패가 정말 사라지는지는 **아직 관찰 중**이다. 재시작 후 재검증되는 라이브 작업의 결과를 지켜봐야 확증된다.
- 검사기가 잡은 진짜 결함들(로그 누출, 계약 미강제, 세션 경합)은 이번에 손대지 않았다. 그것은 구현 쪽이 고칠 몫이며, 검사가 계속 잡는 것이 정상이다.

## 결과·검증 (Outcome)
서버 타입 검사가 통과했고, 범위 검사 관련 단위 테스트 5개와 방금 격리한 테스트 5개가 모두 통과한다. 상시 서비스는 자연 종료가 오지 않아 재시작으로 반영했으며, 재가동 후 서버 상태 점검이 정상이었다. 이번 세션의 커밋 4개를 원격 `main`에 올렸다. 다만 라이브 재검증에서 억울한 실패가 실제로 줄어드는지는 **미검증**으로, 관찰이 남아 있다.
