# Worklog — 병렬로 추가한 목표가 소리 없이 사라지던 문제 · 2026-07-12

## 요약 (TL;DR)
- **문제**: 두 프로젝트에서 목표를 한꺼번에 추가했는데, 한쪽 프로젝트의 목표가 만들어지지 않고 사라졌다.
- **한 일**: 목표가 사라진 원인을 찾아 화면 쪽 저장 로직을 고쳤고, 자동 진행이 멈추던 관련 문제 두 가지도 함께 정리했다.
- **지금**: 세 가지 수정이 라이브 서비스에 반영됐고, 자동 테스트와 실제 서비스 동작으로 확인했다.
- **남은 결정**: 없음.

## 한 일 (What we did)

사용자가 "여러 프로젝트에서 목표를 동시에 추가하면 하나가 사라진다"고 신고했다. 실제 서비스의 데이터베이스와 로그를 직접 뒤져보니, `crewdeck`과 `proof` 두 프로젝트에서 벌어진 일이었다. AI가 목표 후보를 제안하는 과정은 두 프로젝트 모두 정상 실행됐는데, `proof` 쪽 목표만 데이터베이스에 끝내 저장되지 않았다. 서버의 목표 저장은 단순 삽입이라 조용히 누락될 수 없으므로, 원인을 화면(프론트엔드) 쪽으로 좁혔다.

원인은 AI 제안 목록에서 **여러 개를 골라 한 번에 추가하는 경로**에 있었다. 이 처리기는 사용자가 체크한 순번을 제안 목록에서 그대로 꺼내 쓰는데, 오류를 감싸는 장치가 전혀 없었다. 프로젝트를 오가는 동안 제안 목록이 바뀌면 체크해둔 순번이 빈 항목을 가리키게 되고, 그 빈 항목의 제목을 읽는 순간 오류가 나면서 반복문 전체가 중단됐다. 사용자에게는 아무 안내도 없이 고른 목표가 통째로 사라졌다. 빈 항목을 걸러내고, 실패해도 나머지를 살리며, 성공 안내와 실패 안내를 확실히 띄우도록 고쳤다.

조사 중 별개의 문제도 드러났다. 기획서(목표를 실행 가능한 설계로 정리한 문서) 승인을 기다리는 목표를 스케줄러가 1초마다 다시 집어 "승인 대기"만 반복 기록하는 헛돎(busy-loop)이었다. 수정을 만들었지만, 이 세션 도중 다른 세션이 이미 `main`에 다른 방식(반자동·완전자동에서 기획서 자동 승인)으로 같은 문제를 해결해 둔 것을 발견했다. 내 방식은 그 자동 승인이 목표를 집어 승인하는 길을 막아 기존 테스트를 깨뜨렸으므로 철회했다.

다만 자동 승인에도 빈틈이 남아 있었다. 기획서가 승인 검증을 통과하지 못하면(예: AI가 항목을 빠뜨린 불완전한 기획서 생성) 그 목표는 여전히 헛돌며 뒤 목표의 준비까지 막았다. 이 잔여 문제를 마지막으로 고쳤다. 검증에 실패한 기획서 버전을 기억해 두고, 스케줄러가 그 목표를 건너뛰어 다음 목표를 준비하게 했다. 사람이 기획서를 고쳐 새 버전이 생기면 자동으로 다시 시도된다.

## 배운 것 (Learned)

- 여러 Claude 세션이 같은 `main`을 공유하면 세션 도중 `main`이 갈라진다. 이번 한 세션에서만 두 번 벌어졌다. 그래서 코드 작업은 항상 worktree(작업을 격리하는 별도 체크아웃)로 분리해야 한다.
- 화면에서 "목표가 사라졌다"는 대개 백엔드 데이터 유실이 아니라, 오류를 감싸지 않은 비동기 반복문의 조용한 중단이다. 서버 저장이 단순 삽입이면 조용히 누락될 수 없다는 점이 원인 범위를 좁히는 첫 단서가 됐다.
- 두 세션이 같은 버그를 다르게 고쳤다면, 보완되는 쪽만 남기고 충돌하는 접근은 되돌리는 편이 맞다. 억지 병존은 상대의 테스트를 깨뜨린다.

## 세션 흐름 (How it moved)

1. 사용자 신고를 받고 라이브 데이터베이스와 로그로 `crewdeck`과 `proof` 두 프로젝트에서 벌어진 일을 확인했다.
2. `proof` 목표가 데이터베이스에 아예 없다는 사실에서 원인을 화면 생성 경로로 좁혔다.
3. 사용자에게 정확한 조작 순서를 물어, AI 제안에서 여러 개를 골라 추가한 흐름임을 확정했다.
4. 화면 쪽 유실을 고치고, 수정 전 상태를 재현하는 회귀 테스트를 붙였다.
5. 헛돎을 발견했지만 `main`이 이미 자동 승인으로 해결한 것을 확인하고, 충돌하는 내 수정을 철회했다.
6. 자동 승인이 검증에 실패하는 잔여 문제를 고치고 회귀 테스트를 붙였다.
7. 큐를 비우려다 품질 검증이 길어져 강제로 재시작했고, 라이브에서 확인한 뒤 worktree를 정리했다.

## 남긴 것 (Left open)

- 없음. 자동 승인 검증에 실패한 목표는 이제 활동 로그에 "수동 검토 필요"로 한 번 표면화되므로, 사람이 기획서를 고치면 그대로 자동 재개된다.

## 결과·검증 (Outcome)

세 가지 수정이 `main`에 반영되어 라이브 서비스에 배포됐다. `/gate`는 실행하지 않았고, 검증은 자동 테스트와 실제 서비스 관찰로 했다.

자동 테스트는 서버 단위 테스트 506개와 대시보드 테스트 25개가 모두 통과했고, 서버와 대시보드 타입 검사도 통과했다. 새로 붙인 회귀 테스트 두 개는 모두 수정 전에는 버그를 재현하며 실패하고 수정 후 통과하는 것을 확인했다.

라이브에서는 재시작 이후 스케줄러의 "승인 대기" 반복 로그가 **0건**이었다. 멈춰 있던 목표 하나가 자동 승인을 거쳐 태스크 8개로 분할되어 실행에 들어갔다. 강제 종료됐던 검증 태스크는 유령 복구로 정상 완료됐다.
