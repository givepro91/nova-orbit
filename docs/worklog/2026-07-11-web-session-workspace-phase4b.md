# Worklog — 웹 세션 워크스페이스 Phase 4b(끼어들기 + 코드 되돌리기) 구현·배포 · 2026-07-11

## 요약 (TL;DR)
- **문제**: 대시보드에서 에이전트와 대화할 때, 이미 일하는 중이면 말을 걸어도 지금 하던 걸 멈추고 새 지시를 받게 할 방법이 없었고, 에이전트가 코드를 잘못 고쳤을 때 "방금 전 상태로 되돌리기"가 없었다.
- **한 일**: 실행 중 끼어들어 방향을 바꾸는 기능과, 대화 각 턴 시작 시점의 코드를 안전하게 스냅샷해 되돌리는 기능을 붙였다.
- **지금**: 두 기능 모두 라이브 서비스에 배포돼 동작한다. 코드 되돌리기의 핵심 로직은 실제 git 저장소로 자동 테스트를 만들어 통과를 확인했고, 화면에서의 실사용은 아직 미검증이다.
- **남은 결정**: 라이브 자동개발 에이전트가 배포 대상과 같은 폴더에서 돌아 배포 때마다 충돌한다 — 이 구조를 분리할지 결정이 필요하다.

## 한 일 (What we did)

이번 세션은 "웹 세션 워크스페이스" 4단계의 마지막 조각인 개입 모델을 완성했다. 3단계까지는 실행 중인 에이전트에게 메시지를 큐에 쌓거나(Phase 4a) 중단(Esc)할 수 있었는데, 여기에 **끼어들기(steer)** 를 더했다. 실행 중에 `⌘⏎`를 누르면 지금 턴을 멈추고, 방금 입력한 메시지를 이어서 바로 다음 턴으로 재개한다. Claude와 Codex 두 백엔드 모두에서 동작한다.

두 번째로 **코드 체크포인트**를 붙였다. 대화의 각 턴이 시작되기 직전 작업 폴더의 코드 상태를 스냅샷으로 떠 두고, 에이전트가 코드를 망가뜨렸을 때 하단의 "↩ 되돌리기" 칩으로 그 시점 코드로 복원한다. 편집은 되돌리되 새로 만든 파일은 지우지 않는 안전 우선 방식이다.

구현은 병렬 세션 규약대로 격리된 git worktree에서 진행하고, 별도 포트로 인스턴스를 띄워 라이브(7200)를 건드리지 않고 검증했다. 마지막에 main으로 병합하고 drain 절차로 라이브에 배포했다.

## 결정과 왜 (Decisions & why)

| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 체크포인트를 `git stash`가 아니라 임시 인덱스로 뜨는 비파괴 스냅샷(`snapshotWorkdir`)으로 구현 | 계획서 초안의 `git stash push` 재사용 | 대화·소환 세션은 격리된 goal 작업공간이 아니라 **실제 프로젝트 폴더에서 코드를 편집**한다. `git stash`는 작업 트리를 실제로 비웠다 되돌리므로, 사용자가 손대던 미커밋 작업을 덮어 유실시킬 수 있다. 임시 인덱스(`GIT_INDEX_FILE`)로 커밋 객체만 만들면 작업 트리·인덱스·stash 스택을 전혀 건드리지 않는다. |
| 되돌리기는 편집만 원복하고 새 파일은 삭제하지 않음 | 스냅샷 시점으로 완전 초기화(신규 파일 삭제 포함) | 반복 수정이 상태를 악화시키는 Bolt식 Try-to-Fix 안티패턴을 피하려고 "되돌리기"를 우선 노출했다. 그런데 되돌리기가 파일을 지우기까지 하면 그 자체가 파괴적이라, 편집 복원에 국한하고 삭제는 사람 몫으로 남겼다. |
| steer를 새 엔드포인트 없이 기존 큐+drain에 얹음 | 별도 steer 전송 경로 신설 | 메시지를 큐 맨 앞에 넣고 현재 턴만 kill하면, 중단된 턴의 마무리 단계가 이미 있는 큐 배출 로직을 돌려 그 메시지를 다음 턴으로 이어보낸다. 별도 전송 경로를 두면 같은 메시지가 두 번 나갈 위험이 생긴다. |
| 중단된 턴에 `interrupted` 표식을 새로 둠 | 기존 kill 경로(실패로 처리) 그대로 사용 | 기존 종료 처리는 SIGTERM에도 "실패" 상태를 냈다. steer는 정상 개입인데 실패 배지가 뜨면 오해를 부르고, 무엇보다 재개용 세션 식별자를 잃어 다음 턴이 대화 맥락을 이어받지 못한다. 표식을 두어 실패로 취급하지 않고 재개 식별자를 보존한다. |

## 배운 것 (Learned)

- **tsup `splitting: true`로 빌드된 서버 번들은 함수 정의가 공유 청크로 흩어진다.** 배포 검증에서 `dist/server/index.js` 하나만 grep하면 호출부만 보이고 정의가 없어 "번들이 불완전하다"는 오판을 하게 된다. 번들 완전성은 반드시 dist 전체 청크를 대상으로 확인해야 한다.
- **자기 자신을 개발하는 라이브 자동개발 에이전트는 배포와 정면 충돌한다.** 그 에이전트가 배포 대상과 같은 `main` 체크아웃에서 돌면, 사람이 배포하는 순간 main이 움직여 병합이 막히고, 자동개발 쪽 git 정리 작업이 사람의 작업 트리를 덮는다. "라이브 build+restart는 한 세션만 소유"라는 규약의 다른 얼굴이다.

## 세션 흐름 (How it moved)

1. 핸드오프가 짚은 함정(kill이 SIGTERM만 보냄, 종료 핸들러가 중단도 실패로 처리, 되돌리기 함수가 dead-wired)의 라인 번호를 현재 코드에서 먼저 대조했다.
2. 체크포인트가 실제 프로젝트 폴더에서 돈다는 사실을 확인하고, `git stash` 방식의 유실 위험을 사용자에게 보고한 뒤 비파괴 스냅샷으로 방법을 바꿨다.
3. 격리 worktree에서 steer(양 백엔드)와 체크포인트를 구현하고, 스냅샷·복원 로직에 실제 git 저장소 기반 자동 테스트를 붙여 4개 통과를 확인했다.
4. main 병합 중 라이브 자동개발 에이전트가 main을 2커밋 전진시켜 놔서 3-way 병합이 필요했고, 사용자 표시 문구 파일 충돌만 양쪽 유지로 해소했다.
5. 배포는 drain 절차를 따랐다. 자동 실행 큐를 먼저 멈춰 새 작업 투입을 막고, 전체 빌드 후 서비스를 재시작하고, 마지막에 큐를 다시 켰다. 그다음 신규 기능의 접점이 라이브에 실제로 물렸는지 확인했다.

## 남긴 것 (Left open)

- **화면 실사용 검증 미완료.** steer가 실제 대화에서 턴을 끊고 매끄럽게 재개하는지, 되돌리기 칩이 실편집을 원복하는지는 실제 에이전트 대화가 필요해 이번엔 확인하지 못했다. 자동 테스트로 코어 로직과 배선까지는 관통했다.
- **라이브 자동개발 에이전트의 폴더 분리.** 배포 때마다 같은 충돌이 재발하는 구조라, 자동개발 대상 프로젝트를 별도 체크아웃·데이터로 떼는 방안을 검토할 만하다. 이번엔 관찰만 남긴다.

## 결과·검증 (Outcome)

Phase 4b의 두 기능은 main(`b42e119`)에 병합돼 라이브에 배포됐다. 코드 되돌리기의 스냅샷·복원 로직은 실제 git 저장소로 자동 테스트 4개(비파괴 캡처, 편집만 복원·신규 파일 보존, 미커밋 작업 복원, git 아님 폴백)를 새로 만들어 전부 통과를 확인했고, 전체 단위 테스트도 이 4개를 포함해 통과했다(기존부터 실패하던 스쿼시 커밋 관련 2건은 이 변경과 무관하며 그대로 남아 있다). 타입 검사는 서버·대시보드 모두 통과하고, 새로 만든 화면·서버 파일에는 새 린트 경고가 없다. 배포 후 라이브에서 되돌리기·끼어들기 접점이 정상 응답하는 것까지 확인했으나, 화면에서의 실사용은 위에 적은 대로 미검증이다.
