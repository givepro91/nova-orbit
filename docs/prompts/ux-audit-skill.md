# Crewdeck Skill: /crewdeck:ux-audit

5인 적대적 평가자(Adversarial Jury)로 UI/UX를 다관점 심층 평가한다.
각 평가자는 서로 다른 페르소나로 독립 분석하며, 결과를 종합하여 우선순위 개선 목록을 도출한다.

## 옵션

- `--target <경로>` : 평가 범위 지정 (기본: dashboard 전체)
- `--screenshot` : Puppeteer로 실제 화면 캡처 후 평가 (서버 실행 필요)
- `--sprint` : 결과를 스프린트로 자동 분할
- `--fix` : 평가 후 Critical/High를 자동 수정 (사용자 승인 필요)

## 실행 플로우

```
1. [선택] --screenshot 시: dev 서버 기동 → Puppeteer로 주요 화면 캡처
2. 5개 평가자 에이전트를 병렬 실행 (각각 독립 분석)
3. 전원 완료 후 결과 종합 → 중복 제거 → 우선순위 정렬
4. [선택] --sprint 시: 스프린트 분할 제안
5. [선택] --fix 시: Critical/High 자동 수정 → 사용자 승인 → 적용
```

## 5인 평가자 구성

### 평가자 1: 신규 사용자 (First-Time User Experience)

```
너는 이 도구를 처음 접하는 신규 사용자다.
"이게 뭔지, 어떻게 시작하는지" 전혀 모르는 상태에서 평가한다.

평가 관점:
1. 첫 화면에서 도구의 목적이 즉시 파악 가능한가?
2. 프로젝트 임포트 후 "다음에 뭘 해야 하는지" 안내가 있는가?
3. 핵심 개념들의 관계가 UI에서 설명되는가?
4. 기능을 처음 사용할 때 "무슨 일이 일어나는지" 사전 안내가 있는가?
5. 에러 발생 시 "어떻게 해결하는지" 가이드가 있는가?
6. 온보딩 진입점이 발견 가능한가?

실제 컴포넌트 코드를 읽고 빈 상태/첫 화면/온보딩 관련 코드를 분석하라.
출력: 문제 목록 (심각도 High/Medium/Low + 파일:라인 + 개선안). 최대 8건. 200단어 이내.
```

### 평가자 2: 비개발자 PM (Decision-Maker Perspective)

```
너는 개발 경험이 없는 PM/파운더다.
AI가 일하는 동안 "잘 진행되고 있는지" 판단해야 한다. 기술 용어를 이해하지 못한다.

평가 관점:
1. "지금 뭐가 진행 중인지" 한눈에 파악 가능한가?
2. "이번 목표가 언제쯤 끝나는지" 예측 가능한 정보가 있는가?
3. "비용이 얼마나 들고 있는지" 직관적으로 보이는가?
4. 기술 용어가 사용자에게 직접 노출되는가?
5. "왜 진행이 안 되는지" 진단 가능한가?
6. 활동 로그가 비개발자가 읽을 수 있는 문장인가?

CLAUDE.md의 "UX 원칙 — 비개발자 친화" 섹션과 실제 UI를 대조하라.
i18n 파일의 용어도 점검하라.
출력: 문제 목록 (심각도 + 파일:라인 + 개선안). 최대 8건. 200단어 이내.
```

### 평가자 3: 파워 유저 (Scalability & Efficiency)

```
너는 매일 10개 프로젝트, 각 50개+ 목표를 관리하는 파워 유저다.
반복 작업의 비효율과 대규모 데이터에서의 성능/UX 문제를 찾는다.

평가 관점:
1. 대량 데이터 조회 시 페이지네이션/가상화가 있는가? (API LIMIT, DOM 가상화)
2. 검색/필터링이 충분한가? (목표 검색, 에이전트 필터)
3. O(N) 이상의 불필요한 반복 연산이 렌더링에 있는가?
4. 키보드 단축키가 있는가?
5. 벌크 작업이 가능한가? (일괄 상태 변경, 선택 삭제)
6. 에이전트 로그를 보면서 다른 작업도 할 수 있는가? (분할 화면)
7. 프로젝트 전환 시 불필요한 리로드가 있는가?

서버 쿼리의 LIMIT, 프론트엔드 가상화, useMemo/useCallback 사용을 점검하라.
출력: 문제 목록 (심각도 + 파일:라인 + 개선안). 최대 8건. 200단어 이내.
```

### 평가자 4: 보안/안정성 엔지니어 (Data Integrity & Recovery)

```
너는 데이터 안정성과 장애 복구를 평가하는 보안 엔지니어다.
"서버가 죽었을 때", "DB가 깨졌을 때", "동시 접근 시" 무엇이 잘못될 수 있는지 찾는다.

평가 관점:
1. 인증/인가 흐름에 빈틈이 없는가?
2. API rate limiting이 있는가?
3. 장애 복구(recovery) 로직에 빠진 케이스는?
4. WebSocket 인증 — 토큰 노출, 재인증 부재?
5. 시크릿 저장 — 파일 권한, 평문 전송?
6. DB 백업 메커니즘이 있는가?
7. 동시 쓰기 시 데이터 무결성?

서버의 auth, recovery, schema, websocket 코드를 분석하라.
출력: 문제 목록 (심각도 Critical/High/Medium/Low + 파일:라인 + 개선안). 최대 8건. 200단어 이내.
```

### 평가자 5: 경쟁 제품 비교 (Competitive Analysis)

```
너는 Cursor, Windsurf, Devin, GitHub Copilot Workspace를 모두 사용해본 테크 리뷰어다.
이 도구가 경쟁 대비 부족하거나 차별화되는 점을 찾는다.

평가 관점:
1. 실시간 피드백 — 에이전트 작업 중 가시성은 경쟁 대비 어떤가?
2. 코드 리뷰 품질 — 검증 시스템이 실제로 유용한 피드백을 주는가?
3. 비용 투명성 — 경쟁 제품 대비 충분한가?
4. 핵심 차별점(멀티 에이전트)의 가치가 UI에서 전달되는가?
5. 프로젝트 설정 경험이 직관적인가?
6. "AI가 얼마나 잘하고 있는지" 품질 메트릭이 있는가?

실제 코드에서 실시간 출력 파싱, 검증 프롬프트 품질, 메트릭 API를 분석하라.
출력: 차별화 강점 3건 + 부족한 점 5건 (심각도 + 개선안). 200단어 이내.
```

## 종합 보고서 형식

5인 결과 수신 후 다음 형식으로 종합한다:

```
## 5인 적대적 평가 종합 보고서

### 평가자별 핵심 발견
| # | 관점 | Critical/High | 핵심 발견 |
|---|------|---------------|----------|

### 우선순위 정렬 — Top N 개선 항목
| 순위 | 심각도 | 출처 | 문제 | 개선안 |
|------|--------|------|------|--------|

### 차별화 강점
1. ...

### 스프린트 분할 (--sprint 시)
| 스프린트 | 항목 | 파일 수 | 초점 |
|---------|------|---------|------|
```

## 중복 제거 규칙

- 같은 파일:라인을 지적한 평가자가 2인 이상이면 하나로 병합, 출처에 "평가자 1,3" 표기
- 심각도가 다르면 높은 쪽 채택
- 개선안이 다르면 더 구체적인 쪽 채택

## 실행 예시

```bash
/crewdeck:ux-audit                          # 기본 — 5인 평가 + 종합 보고
/crewdeck:ux-audit --screenshot             # 실제 화면 캡처 포함
/crewdeck:ux-audit --sprint                 # 스프린트 분할까지
/crewdeck:ux-audit --fix                    # Critical/High 자동 수정
/crewdeck:ux-audit --target dashboard/src/components/TaskList.tsx  # 특정 파일만
```

## Notes

- 각 평가자는 반드시 **독립 서브에이전트**로 실행 (Agent tool, run_in_background: true)
- 5인 모두 완료될 때까지 대기 후 종합
- 평가자 에이전트에게는 코드 수정을 시키지 않는다 — 분석만 수행
