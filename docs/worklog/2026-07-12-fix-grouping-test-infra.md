# Worklog — 수정 작업 그룹핑과 테스트 인프라 정비 · 2026-07-12

## 요약 (TL;DR)
- **문제**: AI 팀이 품질 검사에서 실패하면 "수정" 작업을 자동으로 만들어 붙인다. 그런데 이 수정 작업이 원래 작업과 화면에서 따로 흩어져 보여, 사용자가 "이건 원래 작업의 하위 작업 아니냐"고 물으면서 시작됐다.
- **한 일**: 수정 작업을 원래 작업 아래에 접어 묶어 보여주도록 화면을 고쳤고, 그 과정에서 드러난 자동화 테스트 설정의 문제들까지 함께 정리했다.
- **지금**: 실서비스 화면에서 수정 작업이 원래 작업 밑에 묶여 보이고, 대시보드 테스트가 실제로 돌아가며, 개발용 테스트 명령이 5분 45초에서 8초로 줄었다. 모두 확인 완료.
- **남은 결정**: 없음. 무거운 통합 테스트는 별도 명령으로 분리돼 릴리스 전에만 돌린다.

## 한 일 (What we did)
품질 게이트(구현 결과를 다른 세션이 채점하는 검증 단계)가 실패를 내면, 엔진은 문제 항목마다 "[수정]" 작업을 새로 만들어 원래 작업이 통과할 때까지 수정과 재검증을 반복한다. 이 수정과 원본의 관계는 이미 `verification_issue_tasks`(어떤 검증의 어떤 문제가 어떤 수정 작업을 낳았는지 기록하는 표)에 저장돼 있었다. 다만 화면에서는 수정 작업이 원래 작업과 다른 상태 칸에 흩어져, 하나의 덩어리로 읽히지 않았다.

사용자의 직관인 "수정은 원래 작업의 하위"는 맞는 말이었다. 그래서 데이터베이스 구조를 바꾸는 대신, 이미 있는 관계를 화면에만 드러내기로 했다. 백엔드는 수정 작업에 "근본 원본 작업 id"를 읽기 전용 파생값으로 실어주고, 대시보드는 기존의 하위 작업 렌더링을 그대로 재사용해 수정 작업을 원래 작업 밑에 접힌 상태로 묶는다.

이후 실서비스 재배포를 준비하다가 예상치 못한 문제를 만났다. 자동 개발을 돌리던 AI 팀이 대시보드 테스트 파일을 커밋했는데, 그 테스트가 설치되지 않은 도구를 불러 프로덕션 빌드를 깨뜨리고 있었다. 그대로 재배포하면 서비스가 빈 화면으로 방치될 위험이었다. 급한 불로 빌드에서 테스트를 제외한 뒤, 이어서 그 테스트들이 실제로 돌아가도록 실행 환경을 제대로 갖췄다.

마지막으로 서버 테스트가 5분 45초나 걸리는 이유를 조사했다. 시간의 대부분은 실제 git 작업을 수행하는 통합 테스트 몇 개가 먹고 있었다. 이 테스트들은 가장 위험한 코드를 지키므로 지울 게 아니라 분리하는 것이 답이었다. 평소 개발용 명령은 빠른 것만 8초에 돌리고, 무거운 것은 별도 명령으로 뺐다.

## 결정과 왜 (Decisions & why)
| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 수정과 원본의 관계를 화면 그룹핑으로만 표현 | `tasks.parent_task_id`에 관계를 저장하는 스키마 변경 | 관계는 이미 `verification_issue_tasks`에 문제·라운드 단위로 더 정밀하게 있다. `parent_task_id`는 이미 위임 하위작업이라는 다른 용도의 주인이 있어, 여기에 얹으면 두 계층이 한 컬럼에서 섞인다. |
| 깨진 대시보드 테스트를 프로덕션 빌드에서 제외 | 그대로 두고 재배포 강행 | 프로덕션 빌드는 원래 테스트를 컴파일하지 않아야 한다. 빌드가 깨진 채 배포하면 정적 파일이 비워진 상태로 서비스가 내려간다. |
| 대시보드 테스트를 대시보드 패키지에서만 실행 | 루트 테스트 목록에 포함 유지 | 테스트 실행에 필요한 도구가 대시보드 폴더에만 설치돼 있어, 루트에서 돌리면 도구를 못 찾아 실패한다. 패키지가 자기 테스트를 책임지는 구조가 맞다. |
| 느린 통합 테스트를 지우지 않고 분리 | 삭제해서 시간 절약 | 느린 테스트가 지키는 것이 작업·커밋 유실 위험이 크고 손으로 검증이 거의 불가능한 코드다. 5분 아끼려 지우면 실제 사고를 놓친다. |

## 배운 것 (Learned)
- 자동 개발이 여러 goal을 합쳐 커밋하면서, 빌드를 깨는 코드를 메인에 올릴 수 있다. 실서비스 재배포 전에는 항상 격리된 폴더에서 빌드가 실제로 성공하는지 먼저 확인한다.
- 관계 데이터가 이미 있으면, 스키마를 늘리기 전에 그 데이터를 화면에 드러내는 것으로 충분한 경우가 많다.
- 테스트가 느린 것이 곧 가치가 없다는 뜻은 아니다. 느린 이유와 그 테스트가 막아주는 위험을 먼저 본 다음, 지울지 분리할지 정한다.

## 세션 흐름 (How it moved)
1. 수정 작업이 어떻게 만들어지는지 설명하고, "그러면 원래 작업의 하위 아니냐"는 사용자 질문에서 출발했다.
2. 스키마 대신 화면 그룹핑으로 방향을 정하고, 격리 인스턴스에서 스크린샷으로 확인한 뒤 실서비스에 반영했다.
3. 재배포를 준비하다가, 자동 개발이 올린 테스트 파일이 프로덕션 빌드를 깨뜨리는 것을 발견하고 빌드에서 제외했다.
4. 대시보드 테스트 실행 환경을 갖춰 테스트 23개를 실제로 통과시켰다.
5. 테스트 타입 검사까지 켜서, 커밋 전 검사 단계가 테스트의 타입 오류도 막게 했다.
6. 서버 테스트가 느린 원인을 조사하고, 빠른 테스트와 무거운 테스트를 분리했다.

## 남긴 것 (Left open)
- 서버 무거운 통합 테스트(약 5분 30초, `npm run test:e2e`)를 분리한 뒤 전량을 다시 실행하지는 않았다. 앞선 진단 실행에서 동일 설정으로 전부 통과했고, 분리 후에는 선택되는 파일 목록만 확인했다. 릴리스 전에 한 번 전량 실행을 권한다.

## 결과·검증 (Outcome)
- 수정 작업 그룹핑은 실서비스에 반영됐다. 수정 작업 66개가 모두 올바른 원본으로 묶이는 것을 API 응답으로 확인했고, 원래 작업 밑에 수정 작업이 접혀 나오는 화면은 격리 인스턴스에서 스크린샷으로 확인했다.
- 테스트 정비 결과, 대시보드 테스트 23개가 통과하고 타입 검사도 통과한다. 개발용 `npm test`는 서버 빠른 테스트 501개를 8초에 통과한다.
- 이번 정비는 실행 코드가 아니라 개발 도구 설정이라, 실서비스 재배포는 필요하지 않았다.
