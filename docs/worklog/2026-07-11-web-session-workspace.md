# Worklog — 웹 세션 워크스페이스를 라이브까지 올리고, 상태 문서를 버리다 · 2026-07-11

## 요약 (TL;DR)
- **문제**: 실패한 작업을 손보려면 매번 터미널을 켜고 상황을 다시 설명해야 했다. 그 왕복을 없애는 대화형 기능을 대시보드 안에 완성해야 했다.
- **한 일**: 실패 카드에서 에이전트를 불러 맥락을 자동으로 채워 대화로 고치는 기능을 세 단계에 걸쳐 만들고, 실제 서비스에 올린 뒤, 처음 쓰는 사람을 위한 그림 설명서를 대시보드 안에 붙였다.
- **지금**: 기능이 실서비스에서 동작한다. 화면 흐름과 도움말은 브라우저로 직접 확인했고, 실제 대화가 필요한 두 지점은 아직 눈으로 못 봤다("미검증"으로 남김).
- **남은 결정**: 없음. 다음 단계(실행 중 끼어들기·되돌리기)는 다른 작업자에게 넘겼고, 미검증 두 지점은 실제 대화 한 번이면 확인된다.

## 한 일 (What we did)

지난 세션에서 대화형 세션의 코어(멀티턴 대화·구조화 렌더)는 코드까지 끝났지만 실서비스에서 돌려본 적이 없었다. 이번 세션은 그 검증으로 시작했다. 실제 에이전트에 두 번 말을 걸어 맥락이 이어지는지 봤고, 그 과정에서 답변이 화면에 두 번 찍히는 버그를 잡았다(`75786ba`). 여기까지가 Phase 1의 마무리다.

이어서 세 단계를 새로 만들었다. Phase 2(`4ab8a97`)는 실패한 태스크 카드의 `⚡ 소환` 버튼이다. 누르면 그 목표의 기획서와 작업 공간, 판정, 최근 출력을 대화에 자동으로 주입하고, 무엇을 넣었는지 칩으로 보여준다. Phase 3(`db12441`)은 좁은 채팅 패널을 좌우 2단 워크스페이스로 넓혀, 왼쪽 대화 옆에서 변경 내역과 판정, 출력을 탭으로 본다. Phase 4a(`a337547`)는 에이전트가 일하는 중에도 입력을 막지 않고 큐에 쌓거나 중단할 수 있게 했다.

작업 도중 실제 문제를 하나 겪었다. 같은 코드 폴더를 다른 세션이 동시에 편집하고 있어서, 내 변경과 남의 미완성 변경이 한 커밋에 섞일 뻔했다. 그래서 중간부터 `git worktree`(같은 저장소를 별도 폴더로 복제해 격리하는 git 기능)로 분리해 작업했고, 완성한 뒤에만 본류에 합쳤다.

마지막으로 이 기능의 사용법을 설명서로 만들었다. 실제 화면을 그대로 흉내 낸 그림 설명을 먼저 공유용 웹페이지로 만들고, 같은 내용을 대시보드 안의 도움말 모달(`?` 버튼)로도 붙였다(`64bbf38`).

## 배운 것 (Learned)
- 사람이 여러 세션을 동시에 돌릴 땐 세션마다 `git worktree`로 격리해야 한다. 같은 폴더를 공유하면 한 세션의 되돌리기·빌드가 다른 세션의 저장 안 된 변경을 덮어 실제로 유실된다. 이 교훈은 이번 세션 중 다른 작업자가 이미 `AGENTS.md`의 규약으로 굳혔다.
- 화면(대시보드)만 바뀌는 변경은 서버를 재시작하지 않고 대시보드만 다시 빌드해 무중단으로 반영할 수 있다.
- 실행 중인 대화 세션을 중단·재개하는 기능은 겉보기보다 계약이 까다롭다. "죽였다"고 믿는 코드가 다른 키의 세션을 보고 있거나, 죽여도 성공으로 응답되는 경로가 숨어 있다. 만들기 전에 kill·resume 계약을 반드시 코드로 확인해야 한다.

## 세션 흐름 (How it moved)
1. 지난 세션에서 코드만 끝난 대화형 세션 코어를 실서비스에서 검증하고, 답변 중복 렌더 버그를 고쳤다.
2. 세션 간 충돌을 반복시키던 상태 문서 `ROADMAP.md`를 폐기하고, 이를 참조하던 지침과 개인 메모리를 정리했다.
3. Phase 2와 3, 4a를 별도 워크트리에 격리해 구현하고, 각 단계를 타입 검사로 검증한 뒤 본류에 합쳤다.
4. 완성한 세 단계를 원격과 라이브 서비스 양쪽에 반영하고, 화면 흐름과 신규 API를 실제로 확인했다.
5. 사용법을 공유용 웹페이지와 대시보드 도움말 모달로 만들어 라이브에 무중단 반영했다.

## 결과·검증 (Outcome)
서버·대시보드 타입 검사가 모두 통과했고, 새 코드가 추가한 린트 경고는 0이다. 별도 포트에 격리 인스턴스를 띄워 Phase 2와 3의 화면 흐름(소환 버튼, 2단 워크스페이스, 4개 탭, 변경 내역 조회)을 브라우저로 관통 확인했다. 신규 API 세 개(`chat/abort`, `goals/:id/diff`, `goals/:id/files`)의 응답도 실서비스에서 직접 호출해 확인했다. 도움말 모달은 라이브 대시보드에서 열어 모든 구획이 렌더되는지 봤다. `/gate`는 이번 세션에 돌리지 않았다. 최종 상태는 원격과 로컬, 라이브가 모두 같은 커밋(`64bbf38`)이다.
