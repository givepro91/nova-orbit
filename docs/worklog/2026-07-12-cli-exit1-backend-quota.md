# Worklog — "Agent CLI exited with code 1" 빨간 알림의 정체 · 2026-07-12

## 요약 (TL;DR)
- **문제**: 대시보드에 "작업 실행이 코드 1로 실패했다"는 빨간 알림이 계속 떴는데, 정작 왜 실패했는지는 알려주지 않아 원인을 알 수 없었다.
- **한 일**: 실패의 진짜 이유가 도구 자체 버그가 아니라 AI 백엔드(Claude·Codex)의 사용 한도 소진이었음을 밝히고, 그 이유가 알림에 그대로 보이도록 하고 한쪽이 막히면 다른 백엔드로 자동 넘어가게 고쳤다.
- **지금**: 자동 테스트는 모두 통과했고 라이브 서비스에 두 번 배포했다. 다만 한도 상황은 그새 풀려버려, "한도에 걸렸을 때 실제로 잘 넘어가는지"는 아직 실환경에서 확인하지 못했다(unverified).
- **남은 결정**: 다음에 실제로 한도에 걸릴 때 알림이 이유를 보여주고 자동 전환이 도는지 관찰하는 것만 남았다.

## 한 일 (What we did)
대시보드에 뜬 빨간 알림은 한 화면에 여러 개가 겹쳐 쌓일 만큼 반복됐고, 문구는 하나같이 `Agent CLI exited with code 1`뿐이었다. 이 도구는 실제 코드 작업을 외부 AI CLI(Claude Code, Codex)에게 하위 프로세스로 시킨다. 그 프로세스가 종료코드 1로 죽었다는 사실만 전달되고 이유는 비어 있었다. 로그를 봐도 표준 에러 출력(`stderr`)이 빈 채 종료코드만 1이었다.

이유는 표준 에러가 아니라 **표준 출력에 숨어 있었다**. Claude Code는 세션 트랜스크립트(각 세션의 대화 기록 파일)에 `isApiErrorMessage`로 표시된 메시지를 남기는데, 거기서 진짜 문구가 나왔다. `You've hit your session limit · resets 10:50pm`(구독 세션 한도 소진)과 `Your organization has disabled Claude subscription access for Claude Code`(조직이 구독 접근을 잠시 차단)였다. 즉 도구의 버그가 아니라, 구독이 그 시간대에 한도에 걸렸던 것이다.

발생 경로도 드러났다. 이 프로젝트는 구현을 기본적으로 Codex로 돌리는데, 검증이 실패하면 자동수정을 일부러 반대 백엔드인 Claude로 넘겨 다른 모델에게 고칠 기회를 준다. 그 자동수정 세션이 Claude 구독 한도에 걸려 죽으면서 빨간 알림이 됐다. 문제는 이 자동수정 경로가 실패를 위로 던지지 않고 알림만 띄운 뒤 재검증으로 넘어가, 이미 존재하던 Codex 자동 전환 장치를 타지 못했다는 점이다.

1차로 두 가지를 고쳤다(자세한 변경은 커밋 `13c8416`). 첫째, 한도·차단 문구를 에러 감지·분류에 등록해 알림에 실제 이유가 뜨고 Codex 전환 대상으로 잡히게 했다. 둘째, 자동수정 실패가 코드 결함이 아니라 백엔드 자체 문제일 때는 위로 던져, 이미 검증된 스케줄러의 Codex 재배정이 처리하도록 했다. 배포 후 사용자가 스크린샷으로 알림이 여전히 뜬다고 알려왔고, 이번엔 원인이 달랐다.

이번엔 **Codex도 사용 한도에 걸려** 있었다(`try again at 1:05 AM`). 잠깐이지만 Claude와 Codex가 동시에 소진돼, 넘어갈 백엔드가 없는 상태였다. 그런데도 알림엔 여전히 이유가 없고, 3초 간격으로 재시도가 폭주했다. Codex는 한도 메시지를 표준 에러가 아니라 표준 출력의 구조화 이벤트(`turn.failed`)로 흘리는데, 실패 판정 함수가 종료코드를 먼저 보고 빈 표준 에러를 이유로 써버려 그 메시지를 버렸고, "usage limit"이 한도 신호 목록에 없어 태스크 자체 실패로 오분류돼 대기(백오프) 대신 즉시 재시도가 반복된 것이다. 이것은 1차 수정의 자매 격 빈틈이었다. 2차로 실패 판정 함수가 표준 출력의 이유를 끌어오게 하고 "usage limit"을 한도 신호로 추가했다(커밋 `87a5eb1`).

## 결정과 왜 (Decisions & why)
| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 백엔드 자체 실패는 자동수정 안에서 위로 던져 스케줄러의 기존 Codex 전환에 맡김 | 자동수정 라운드 안에 별도 재시도 로직을 새로 넣음 | 스케줄러에는 이미 한도·소진 시 반대 백엔드로 재배정하는 검증된 장치가 있다. 새 로직을 중복으로 짜는 대신 그 장치가 닿도록 실패를 흘려보내는 편이 코드가 적고 왕복 차단 같은 안전장치를 그대로 재사용한다. |
| 종료코드가 1이어도 표준 출력에 실린 이유를 먼저 끌어와 알림에 표시 | 종료코드만으로 "code 1" 알림을 내던 기존 방식 유지 | 두 백엔드 모두 진짜 에러를 표준 에러가 아니라 표준 출력에 남긴다(Codex는 구조화 이벤트, Claude는 메시지 텍스트). 종료코드를 먼저 보고 빈 표준 에러를 이유로 쓰면 사용자가 볼 유일한 단서가 사라진다. |
| "usage limit"을 한도(rate limit) 신호로 분류 | 태스크 자체 실패로 두고 재시도 예산으로 처리 | 한도로 분류돼야 재시도 대신 리셋까지 대기(백오프)로 전환된다. 태스크 실패로 두면 무고한 작업이 3초 간격 재시도로 예산만 태우고 알림을 도배한다. |
| 라이브 반영을 위해 진행 중 세션을 의도적으로 중단(사용자 승인) | 진행 중 목표가 끝날 때까지 자연 종료를 기다림 | 큐 정지 명령은 새 목표 유입만 막고, 이미 실행 중인 목표(구현·검증·수정이 이어지는 한 흐름)는 못 멈춘다. 30분을 기다려도 끝나지 않아, 손실을 감수하고 세션을 끊는 쪽을 사용자가 택했다(중단된 태스크는 재시작 후 재실행). |

## 배운 것 (Learned)
- CLI 백엔드가 에러를 표준 에러가 아니라 표준 출력의 구조화 데이터로 흘리면, 종료코드를 먼저 보는 실패 판정은 이유를 통째로 버린다. 종료코드보다 출력에 실린 사유를 먼저 읽어야 한다.
- Claude 세션이 코드 1로 죽은 근본 이유는 서버 로그에는 없고, 세션 트랜스크립트 파일의 `isApiErrorMessage` 메시지에서만 확인된다. 하위 프로세스 CLI를 디버깅할 땐 그 CLI의 자체 기록을 봐야 한다.
- 큐 정지는 이미 실행 중인 목표를 멈추지 못한다. "정지 후 활동이 0이 될 때까지 대기"라는 배포 절차는 진행 중 목표가 있으면 무한정 늘어질 수 있다.
- 두 백엔드가 동시에 한도에 걸리는 일이 실제로 일어난다. 그럴 땐 넘어갈 대안이 없으므로 조용히 삼키지 말고 사용자에게 노출하되, 빨간 재시도 폭주가 아니라 리셋까지 대기하는 형태여야 한다.

## 세션 흐름 (How it moved)
1. 빨간 알림 스크린샷을 받고, 에러 문구의 발생 지점과 라이브 로그를 좁혔다.
2. 실패한 Claude 세션 트랜스크립트에서 진짜 이유(구독 세션 한도·조직 접근 차단)를 찾아 근본 원인을 확정했다.
3. 이유가 사라지는 진단 사각지대와, 자동수정 경로가 Codex 전환을 못 타는 빈틈을 확인했다.
4. 1차 수정을 하고 자동 테스트와 타입 검사를 통과시켰다.
5. 진행 중 세션을 중단(사용자 승인)하고 재빌드·재시작한 뒤 커밋·푸시했다.
6. 재발 스크린샷을 받아 이번 원인이 Codex 사용 한도임을, 직접 실행으로 재현해 확인했다.
7. 표준 출력의 이유를 끌어오고 한도 신호를 보강하는 2차 수정을 한 뒤 배포하고, 커밋해서 푸시했다.
8. Codex 한도가 회복돼 태스크가 정상 진행됨을 확인했다. 알림 폭주가 멎은 것은 이 회복 덕이지 수정의 실증은 아님을 구분해 기록했다.

## 남긴 것 (Left open)
- 실환경 검증이 남았다. 수정의 핵심 효과인 "한도에 걸리면 이유가 뜨고 재시도 폭주 대신 대기로 전환"은, 검증을 시도한 시점에 한도가 이미 풀려 재현하지 못했다. 다음에 실제로 한도에 걸릴 때가 유일한 확인 지점이다.
- 두 백엔드가 동시에 소진된 상황은 지금 한도 대기(백오프)로 처리되지만, "두 백엔드 모두 한도"임을 알리는 전용 문구는 만들지 않았다. 현재는 일반 한도 대기 표시로 갈음된다.

## 결과·검증 (Outcome)
타입 검사가 통과하고, 실패 판정·분류 테스트 24개가 모두 통과했으며 전체 자동 테스트 523개도 통과했다. 라이브 서비스는 두 차례 재빌드·재시작해 정상 응답을 확인했고(health ok), 재빌드한 서버 번들에 새 로직이 포함된 것도 확인했다. 이번 세션은 정직성 검증 단계(`/gate`)를 돌리지 않았으므로, 위 결과는 각 단계에서 직접 실행해 관찰한 값이다. 알림 폭주가 멎은 것은 Codex 한도가 회복된 덕이며, 수정 자체의 실환경 동작은 아직 **unverified**로 남는다.
