# Worklog — 멈춘 에이전트 팀을 리뷰어 승인 게이트로 되살리다 · 2026-07-14

## 요약 (TL;DR)
- **문제**: 두 프로젝트의 AI 팀이 하루 넘게 아무 일도 안 하고 멈춰 있었다. 죽은 게 아니라 "사람이 승인해 주세요" 상태에서 계속 기다리고 있었는데, 아무도 승인하지 않아 굳어 있었다.
- **한 일**: 개발 수준의 승인은 다른 AI가 대신 판단해 통과시키고, 제품 방향을 바꾸는 결정만 사람에게 남기도록 승인 절차를 바꿨다.
- **지금**: 라이브 서비스에 반영을 마쳤고, 멈춰 있던 작업 중 재개해도 되는 것은 자동으로 다시 돌기 시작했다. 실제로 한 태스크가 실행에 들어가는 것까지 확인했다.
- **남은 결정**: 자동 수정까지 시도했는데도 검증을 통과 못 한 태스크 3개는 사람이 봐야 한다. 승인할지 다시 작업시킬지 결정이 필요하다.

## 한 일 (What we did)

시작은 "왜 다 멈췄지?"라는 질문이었다. 서버도 스케줄러도 멀쩡히 돌고 있었지만, 두 프로젝트의 모든 목표가 첫 태스크에서 `pending_approval`(사람 승인을 기다리는 상태)에 걸려 있었다. 스케줄러는 실행할 수 있는 태스크를 못 찾은 채 5분마다 "멈춤" 경고만 수만 번 남기고 있었다.

원인을 확정한 뒤, 사용자와 정책을 합의했다. 목표를 태스크로 쪼갠 계획을 사람이 일일이 승인하던 자리를, 별도의 리뷰어 AI가 대신 검토하도록 바꾼다. 리뷰어는 각 태스크를 세 갈래로 판정한다. 무난하면 승인해 실행으로 보내고, 결함이 있으면 반려하고, 메뉴 삭제 같은 제품 방향 결정이면 사람에게 넘긴다.

핵심은 **품질 검증을 우회하지 않는 것**이었다. 크루덱에는 이미 구현과 검증을 다른 세션이 맡는 품질 게이트(Quality Gate)가 있고, 이건 실측으로 유효성이 입증돼 함부로 없애면 안 되는 장치다. 그래서 "계획 승인"과 "검증 결과"를 구분하는 판별 조건을 넣어, 검증에서 나온 승인 대기 태스크는 리뷰어가 절대 건드리지 못하게 막았다.

작업하며 예상 밖의 버그도 하나 잡았다. 큐를 다시 켤 때 승인 대기 태스크를 전부 무차별로 통과시키던 오래된 코드가 있었는데, 이게 검증에 실패한 태스크까지 조용히 실행시키고 있었다. 이번 판별 조건이 그 구멍도 함께 막았다.

## 결정과 왜 (Decisions & why)

| 결정 | 버린 대안 | 왜 |
|------|----------|----|
| 승인 주체를 별도 리뷰어 AI로 둔다 | 승인 단계를 없애고 무조건 통과, 또는 키워드 규칙으로 통과 | 계획을 짠 세션이 자기 계획을 스스로 승인하는 것은 크루덱의 생성-검증 분리 원칙에 어긋난다. 계획을 짠 관리자 역할과 다른 세션이 검토해야 자기 승인을 피한다. |
| 사람 승인 대상(제품 방향성)을 판별할 때 AI가 태스크별로 분류한다 | 삭제·마이그레이션 같은 키워드 규칙, 또는 목표 등록 시 사람이 미리 지정 | 키워드는 표현을 우회하면 놓치고, 목표 단위 지정은 태스크마다 다른 성격을 못 담는다. 계획을 짤 때 맥락이 가장 풍부한 AI가 판단하고, 리뷰어가 최종 확정하는 편이 정확하다. |
| 검증·수정에서 나온 승인 대기는 게이트에서 제외한다 | 모든 승인 대기를 리뷰어가 처리 | 검증 실패로 생긴 승인 대기는 "자동 수정까지 했는데도 통과 못 함"이라는 신호다. 이걸 리뷰어가 자동 승인하면 품질 게이트를 무력화하는 셈이라, 사람이 봐야 할 몫으로 남긴다. |
| 수정 계획에는 리뷰어 게이트를 넣지 않는다 | 원래 계획에 있던 수정 계획 게이트도 구현 | 코드를 읽어 보니 수정은 승인 후 실행되는 게 아니라 이미 자동 실행되고, 승인 대기는 실패한 뒤의 최종 상태였다. 승인을 끼울 자리가 없고, 끼우면 품질 검증을 건너뛰게 된다. |

## 배운 것 (Learned)

- `pending_approval`이라는 하나의 상태에 성격이 다른 두 가지가 섞여 있었다. 하나는 "실행 전 계획 승인"이고 다른 하나는 "검증 실패로 사람이 봐야 하는 것"이다. 자동화를 논할 때는 이 둘을 반드시 갈라서 다뤄야 한다.
- 무거운 통합 테스트가 실패할 때는, 고치기 전에 변경 이전 버전에서 같은 테스트를 돌려 비교부터 해야 한다. 이번 `goal-as-unit` 통합 테스트의 실패 20건은 변경과 무관한, 원래부터 깨져 있던 것이었다.
- 오래된 "일괄 통과" 코드는 조건이 좁혀지지 않으면 안전장치를 조용히 우회한다. 승인·통과 로직을 만질 때는 무엇을 통과시키지 말아야 하는지부터 명시해야 한다.

## 세션 흐름 (How it moved)

1. 로그와 데이터베이스와 코드를 함께 읽어, 멈춤의 원인이 크래시가 아니라 승인 게이트 정체임을 확정했다.
2. 사용자와 정책 두 가지를 합의했다. 승인 주체는 리뷰어 AI로, 사람 승인 대상 판정은 AI 분류로 한다.
3. 탐색과 설계를 병렬 에이전트에 맡겨, 어디에 게이트를 끼울지와 재사용할 기존 패턴을 확정했다.
4. 워크트리(main과 격리된 작업 사본)에서 구현하고, 단계마다 타입 검사를 통과시켰다.
5. 새 로직을 단위 테스트로 검증하고, 통합 테스트 실패는 변경 이전 버전과 비교해 무관함을 가렸다.
6. 큐를 멈추고 다시 켜는 배포 절차로 라이브에 반영하고, 멈춰 있던 작업이 의도대로 갈라지는 것을 확인했다.

## 남긴 것 (Left open)

- 검증에 실패한 태스크 3개는 승인 대기로 남아 있다. 각 검증 실패 내용을 요약해 사람이 판단해야 한다.
- 실제 AI가 계획을 짤 때 "사람 승인 필요" 표시를 제대로 생성하는지, 리뷰어 세션이 실제로 판정을 내리는지는 **직접 관통 검증하지 않았다**. 프롬프트가 어긋나도 안전하게 승인 대기로 남는 폴백이 있어, 라이브에 새로 등록되는 목표에서 자연 관찰하기로 미뤘다.

## 결과·검증 (Outcome)

타입 검사가 통과했고, 빠른 단위 테스트 585개에 새 테스트 5개를 더해 모두 통과했다. 새 테스트는 세 갈래 판정과 품질 게이트 보존, 수동 모드에서의 무동작, 리뷰어 실패 시 안전 폴백을 직접 확인한다.

전체 빌드가 성공했고, 배포 후 라이브 데이터베이스에 새 컬럼 두 개가 실제로 생성된 것과 서버 정상 기동을 확인했다. 재기동 직후 멈춰 있던 작업이 설계대로 갈라졌다. 재개해도 되는 태스크 2개는 자동 승인돼 그중 하나가 실행에 들어갔고, 검증 실패 3개는 승인 대기로 남았다.

무거운 통합 테스트(`goal-as-unit`)는 20건이 실패하지만, 변경 이전 버전에서 돌린 결과와 실패 목록까지 완전히 같았다. 이 실패들은 이번 변경과 무관한, 이전부터 깨져 있던 것이다.
