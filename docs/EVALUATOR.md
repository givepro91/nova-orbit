## 🔴 CRITICAL — 프로젝트를 시작하기 전에 반드시 검증해야 할 것

### 1. ~~Claude Code CLI를 서브프로세스로 제어할 수 있는가?~~ ✅ 해소

**Paperclip 소스 분석으로 검증 완료.**

Paperclip의 `claude_local` 어댑터가 이 패턴을 프로덕션에서 검증함:

- `child_process.spawn`으로 Claude Code CLI 서브프로세스 실행
- `stdin/stdout(stdio)` 파이프로 태스크 전달/결과 수신
- `--add-dir` 플래그로 스킬 파일 주입 (임시 디렉토리 + 심볼릭 링크)
- `sessionBehavior: "resume-or-new"`로 세션 영속성 확보
- 내장 DB(PGlite)로 별도 DB 설정 불필요

**→ PoC 불필요. Paperclip의 claude_local 어댑터 코드를 분석하고 Crewdeck에 맞게 재구현하면 됨.**
**→ Sprint 1에서 Paperclip 소스 분석을 첫 태스크로 진행.**

### 2. Claude Pro 구독 하나로 동시에 몇 세션을 돌릴 수 있는가?

문서는 "에이전트 3개 = Claude Code 세션 3개"라고 가정합니다. 하지만:

- Claude Pro의 동시 세션 제한은? Rate limit은?
- 에이전트 3개가 동시에 작업하면 토큰 소모가 3배. Claude Pro의 사용량 제한에 걸리지 않는가?
- Evaluator 세션까지 포함하면 실질적으로 4-5개 세션이 필요. Pro로 가능한가?
- 불가하다면 Team/Max 플랜이 필수인데, 그러면 타겟 유저(1인 창업자)의 비용 부담이 큼

**문서가 "$0 추가 비용"이라고 하지만, 현실적으로 Claude Pro 사용량을 빠르게 소진시켜서 유저가 플랜을 업그레이드해야 할 수 있습니다.** 이건 숨겨진 비용이고, 유저 경험을 나쁘게 만듭니다.

→ **Action: Claude Pro/Team/Max 각 플랜의 rate limit과 동시 세션 제한을 정확히 확인하고, "에이전트 N개 = 실제 필요 플랜"을 매핑하세요.**

---

### 3. Paperclip 팀이 Quality Gate를 추가하면?

Crewdeck의 핵심 차별점은 "오케스트레이션 + Quality Gate"입니다. 하지만 Paperclip은:

- 44k GitHub 스타, 풀타임 팀
- MIT 오픈소스 → 커뮤니티 기여로 빠르게 기능 추가
- 이미 "Ticket System"과 "Governance"가 있어서, Quality Gate 추가는 자연스러운 확장

**Paperclip이 검증 기능을 6개월 내에 추가할 확률은 높습니다.** 그러면 Crewdeck의 유일한 차별점이 사라집니다.

→ **이것이 의미하는 것: "Quality Gate"만으로는 장기 차별점이 안 됩니다. UX의 압도적 차이, 또는 Claude 네이티브 최적화의 깊이가 진짜 해자(moat)가 되어야 합니다.**

## 🟡 SERIOUS — 문서에서 과소평가된 위험

### 4. Notion 스타일 대시보드는 "간단"하지 않다

문서가 "Sprint 5: 대시보드 UI (5-7일)"이라고 했는데, 이건 **심각하게 과소평가**입니다.

Notion 스타일 UI를 구현하려면:

- 사이드바 네비게이션 (트리 구조, 접기/펼치기)
- 인라인 편집 (클릭 시 바로 수정)
- 드래그 앤 드롭 Kanban
- 실시간 WebSocket 업데이트
- 터미널 미리보기 (xterm.js)
- 레이더 차트 (검증 결과)
- 반응형 레이아웃
- 다크/라이트 모드

이걸 5-7일에 하려면 Shadcn/ui + 기존 Kanban 라이브러리를 최대한 활용해야 하고, 그래도 **2-3주는 걸립니다.** "Notion 느낌"을 내는 것과 "동작하는 대시보드"는 완전히 다른 수준의 작업입니다.

→ **Action: Phase 1에서는 "Notion 느낌"을 포기하고 "동작하는 대시보드"에 집중하세요. 예쁜 건 Phase 2에서. MVP는 기능 증명이 먼저입니다.**

### 5. Phase 1 범위가 너무 넓다

현재 Phase 1에 포함된 것:

- npx 설치 + SQLite
- 프로젝트 CRUD + 로컬 임포트 + GitHub 연결
- 코드베이스 자동 분석 + 에이전트 자동 제안
- Claude Code 세션 관리
- 오케스트레이션 엔진 (목표 → 태스크 분해)
- Nova Quality Gate 포팅
- Notion 스타일 대시보드
- WebSocket 실시간 스트리밍
- 멀티 프로젝트

**이건 MVP가 아니라 v1.0입니다.** 주 15시간+으로 이걸 다 하면 3-6개월은 걸립니다. 그 사이에 모티베이션이 떨어지거나, Paperclip이 비슷한 기능을 내놓을 수 있습니다.

→ **Action: Phase 1을 더 잘라야 합니다. "가장 작은 동작 가능한 루프"를 정의하세요:**

```
진짜 MVP (Phase 0.5):
- npx crewdeck → 서버 기동
- 프로젝트 1개 (새로 생성만, 임포트/GitHub은 Phase 1.5)
- 에이전트 2개 (Coder + Reviewer)
- 태스크 수동 생성 + 수동 할당
- Claude Code 세션 실행 + 결과 표시
- Quality Gate 검증 1회 통과
- 대시보드: 태스크 리스트 + 에이전트 상태 (Kanban은 나중)
```

### 6. Generator-Evaluator가 코드 외 결과물에도 적용 가능한가?

Nova의 5차원 검증(기능, 데이터관통, 설계정합성, 크래프트, 경계값)은 **코드 전용**입니다. 하지만 Crewdeck의 에이전트에는 Marketer, Designer도 있습니다.

- Marketer가 작성한 블로그 포스트를 5차원 검증으로 어떻게 평가하는가?
- Designer가 만든 와이어프레임은?

**코드 외 결과물에 대한 검증 기준이 문서에 없습니다.**

→ **Action: Phase 1에서는 Coder + Reviewer(코드 전용)만 지원하고, 비코드 에이전트 검증은 Phase 2에서 별도 설계하세요. 문서에 이 제한을 명시해야 합니다.**

---

### 7. 수익 모델의 "라이센스 키" 방식은 쉽게 우회된다

MIT 오픈소스 + 로컬 설치형에서 라이센스 키로 Pro 기능을 잠그는 건, 코드를 한 줄 수정하면 우회 가능합니다. 개발자 타겟이라 이걸 할 줄 아는 사람이 대부분입니다.

→ **대안 옵션:**

- **Pro 기능을 클라우드 API로 제공** (검증 로그 분석, 고급 분석 대시보드 등은 서버에서만 동작)
- **Honest paywall**: "무료로 다 쓸 수 있지만, 우리를 지원해주세요" 방식 (Tailwind CSS 모델)
- **Hub를 주 수익으로**: 프리미엄 템플릿은 Hub 서버에서만 다운로드
- 초기에는 수익보다 유저 확보가 먼저이므로, 이 문제는 Phase 3에서 해결해도 됨

## 🟢 GOOD — 문서가 잘한 부분

### 잘된 점

1. **Paperclip 분석 가이드가 구체적이다.** 어디를 참조하고 어디를 참조하지 말아야 하는지 명확.

2. **에이전트 역할 프리셋이 YAML로 정의되어 확장 가능하다.** 커뮤니티가 템플릿을 만들 수 있는 구조.

3. **프로젝트 임포트 + GitHub 연결이 포함되었다.** 기존 프로젝트에 바로 적용할 수 있어야 실사용이 가능.

4. **비용 구조가 건강하다.** 인프라 비용 거의 제로, 마진 95%+. BEP가 매우 낮음.

5. **글로벌 + 국내 이중 전략이 있다.** GitHub/Product Hunt(글로벌) + 디스콰이엇(국내).

## 결론

**이 프로젝트는 할 가치가 있습니다.** 시장(AI 에이전트 오케스트레이션)은 확실히 성장 중이고, Paperclip이 증명한 수요가 있으며, Nova의 Quality Gate는 진짜 차별점입니다.

**하지만 시작 방법이 틀려 있습니다.** 지금 문서대로 Phase 1을 시작하면 3-6개월 뒤에 "거의 다 됐는데 아직 안 됨" 상태에 빠질 가능성이 높습니다.

**올바른 시작:**

```
Day 1-3:    Paperclip 소스 분석 (claude_local 어댑터 중심)
            → docs/paperclip-reference.md 작성
            → Claude Pro 동시 세션 rate limit 실측

Week 1-2:   Phase 0.5 (최소 동작 루프)
            → 프로젝트 1개 + 에이전트 2개 + 태스크 실행 + 검증 1회
            → 대시보드는 가장 기본적인 것만

Week 3-4:   첫 번째 "내가 쓸 수 있는 버전" 완성
            → 본인이 실제 사이드 프로젝트에 사용
            → 불편한 점 기록 → 즉시 개선

Month 2+:   GitHub 공개 + 피드백 수집
```
